# 🤖 AI（Antigravity）用 引き継ぎ・仕様ドキュメント

このファイルは、X（旧Twitter）自動投稿ボット群（さくら、ネコティア、ロト6）のアーキテクチャ、運用上の注意点、および直近のアップデート履歴をまとめたものです。
**新しいAI（Antigravityの別セッション等）が対応を開始する際は、必ず最初にこのファイルを読み込み、現在のシステム状況を把握してください。**

## 1. プロジェクトの全体像とリポジトリ
現在、以下の3つの独立した自動投稿ボットが稼働しています。
1. **恋AIモード（さくら）**: `/Users/user/Desktop/恋AIモードXbot` (repo: `akyng/sakura-renai-xbot`)
2. **ネコティア（Nekotia）**: `/Users/user/Desktop/ネコティア` (repo: `akyng/nekotiabot`)
3. **ロト6（Loto6 Oracle）**: `/Users/user/Desktop/Loto6 Oracle/backend` (repo: `akyng/loto6-auto-poster`)

## 2. コアシステムとアーキテクチャの特徴
*   **投稿の仕組み (Playwright)**:
    *   従来の `tweepy` を用いた X API 経由での投稿は制限が厳しいため、現在はすべて **Playwright を用いたブラウザ自動操作（ヘッドレス Chrome）** へ移行しています。
    *   各プロジェクトの `core/publisher.py` (または `backend/publisher.py`) にブラウザ操作ロジックが集約されています。
*   **インプレッション最大化アルゴリズム（URL分離）**:
    *   Xのシャドウバン・インプレッション低下対策として、メインのポスト（1つ目の投稿）には画像とテキストのみを含め、**プロモーション用のURLリンクは必ず「自分自身へのリプライ（2つ目の投稿）」にぶら下げる** 構造を採用しています。
*   **GitHub Actions での全自動運用**:
    *   定期実行（cron）および手動実行（workflow_dispatch）は GitHub Actions で管理されています。
    *   GitHub Secretsの `X_COOKIE_JSON` にブラウザのログインクッキーを保存し、アクション実行時に Playwright に読み込ませることでログイン状態を維持しています。

## 3. クッキーの更新手順（最重要）
GitHub Actions上で「Xへのログインセッション（クッキー）が失効しています」というエラーが出た場合、以下の手順でクッキーを再抽出してGitHubへ登録する必要があります。
1. **Safari/Chromeでのログイン**:
   対象のアカウント（さくら、ネコティア、ロト6いずれか）で、ユーザーのローカルブラウザ（Google Chrome または Safari）から手動でXにログインする。
2. **クッキーの抽出**:
   ネコティアのフォルダ内にある抽出ツールを実行する。
   `python3 /Users/user/Desktop/ネコティア/extract_my_cookies.py`
   ターミナルの対話プロンプトに従い、アカウントとブラウザを選択する。
3. **GitHub Secretsの更新**:
   抽出されたJSONファイル（`sakura_cookies.json` など）の中身を、対象リポジトリの GitHub Secrets の `X_COOKIE_JSON` に上書き保存する（`gh secret set X_COOKIE_JSON < ...` コマンドを使用可能）。

## 4. 各ボットの固有仕様と状態管理
### A. ネコティア (Nekotia)
*   **情報源**: Google News RSS (`scraper.py`)
*   **重複防止**: `state.json` に過去のURLと次回カテゴリインデックスを保存し、コミットして管理。
*   **工夫点**: ランキング記事などの固定ページでループするのを防ぐため、RSS検索クエリに `when:7d` を付与し「直近7日以内の新しいニュース」のみを収集。さらに、候補の中から「ランダムに」記事を選択してマンネリ化を防止。

### B. さくら (恋AIモード)
*   **情報源**: Gemini APIによる自己生成 (`generator.py`)
*   **重複防止**: プロンプト内で多様なシチュエーション（カフェ、ショッピング、イルミネーションなど）を指定し、画像・テキストともにマンネリ化を防ぐ。

### C. ロト6 (Loto6 Oracle)
*   **情報源**: ロト6公式サイトからのスクレイピング
*   **状態管理**: `cache.json` に「最後に投稿した回号」や「本日のトリビアを投稿した日付」を保存し、重複投稿を防止。
*   **スケジュール**:
    *   水・日の20:30台に「前日分析データ」を投稿。
    *   毎日の12:00台に「トリビア・AI予想宣伝」を投稿。

## 5. 直近のトラブルシュート履歴 (2026年5月時点)
1. **PlaywrightのCookieエラー**: 抽出したクッキーの `secure` や `httpOnly` が数値(0/1)になっているとクラッシュする問題があり、読み込み時にBooleanにサニタイズする処理を追加済み。
2. **ハッシュタグ補完による妨害**: Xの投稿画面でハッシュタグを入力すると補完ドロップダウンが被さり「投稿ボタン」が押せなくなる問題が発生。`page.keyboard.press("Escape")` を送信してオーバーレイを消す処理を追加済み。
3. **リモート環境へのPlaywright導入漏れ**: ローカルでPlaywrightのコードを書いても、`.github/workflows` 内のymlファイルで `playwright install chromium` を実行し `PUBLISH_MODE: browser` を指定しないと動かない。ロト6でこの設定漏れがあり修正済み。
4. **キャッシュのズレ**: ロト6でdryrun（テスト投稿）が成功扱いとなり `cache.json` の日付が進んでしまい、本番投稿がスキップされる事象が発生。`cache.json` を手動でロールバックして解消。
5. **重複・既読記事の再投稿バグ**: ネコティアにおいて、RSSから得た記事がすべて既読の際、過去の記事を再利用したり、固定宣伝URLを繰り返し投稿するフォールバックが原因で重複が発生。`core/scraper.py` に `allow_fallback=False`（デフォルト）を設定して重複時に `NoNewArticlesError` をスローさせ、`main.py` に「新鮮なニュースがあるカテゴリをローテーション順に順次自動探索する」リトライループを実装して、完全な重複回避と運用の自動継続を両立しました。
6. **ローカル最新コードのプッシュ漏れによる重複投稿**: ローカルで実装された最新の重複回避・自動カテゴリ探索コードやバズプロンプト最適化がGitHubにプッシュされておらず、GitHub Actions上で「旧式のフォールバック（`https://investapps.net/`）が有効な古いコード」が動いてしまい、結果として重複URLが投稿されていました。ローカルの最新バグフィックスコードをリモートへ完全同期・プッシュして恒久対応しました。

**[AIへの指示・絶対ルール]**
このファイルを読み込んだAIは、以下の運用ルールを絶対に守って対応してください。

1. **Xのバズるアルゴリズム（シャドウバン対策）への対応**
   メイン投稿（親ポスト）には絶対にURLリンクを含めないこと。URLによるプロモーションや誘導は、必ず「自分自身へのリプライ（子ポスト）」にぶら下げる形で実装してください。
2. **過去の投稿との重複回避（新規性の徹底担保とバズるネタの厳選）**
   同じ内容の投稿や、固定のフォールバック・URL（`https://investapps.net/`等）の投稿が絶対に繰り返されないようにしてください。過去の投稿履歴（`state.json` や `cache.json`、RSSの履歴）を厳格に照合し、**完全に新しい、読者の目を引くタイムリーなバズるネタのみ**を投稿すること。新規ニュースが見つからない場合は、フォールバックURLを投稿するのではなく、その回の自動投稿を安全にスキップ（Exit 0）してください。
3. **さくら（恋AIモード）の画像とシチュエーションの多様化**
   さくらの投稿を生成・修正する際は、画像のマンネリ化を防ぐため、毎回「異なるシチュエーション（カフェ、ショッピング、旅行、季節のイベントなど）」をプロンプトに指定し、新鮮でバリエーション豊かな画像を生成するようにしてください。
4. **ドキュメントの保守**
   システムに変更を加えた場合は、後任 of AIが迷わないようにこのドキュメントを適宜アップデートすること。
